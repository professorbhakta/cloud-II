1. OVERVIEW OF INFRASTRUCTURE AS CODE (IaC)
================================================================================

1.1  DEFINITION (WRITE THIS IN EXAMS)
------------------------------------
Infrastructure as Code (IaC) is the practice of managing and provisioning
computing infrastructure — networks, virtual machines, load balancers,
databases, storage buckets, firewalls, IAM roles, and more — through
machine-readable definition files, rather than through manual processes
such as clicking in a cloud console or typing one-off commands.

  Traditional (manual) way:
    Open GCP / AWS / Azure console → click Create VM → choose size, disk,
    network, tags → hope you remember every setting next time.

  IaC way:
    Write a file that describes the desired VM (and related resources) →
    store the file in Git → run a tool (Terraform / CloudFormation / …) →
    the tool creates or updates the real cloud resources to match the file.

In one line for exams:
  “IaC means treating infrastructure the same way we treat application code —
   versioned, reviewed, tested, and automated.”


1.2  WHAT COUNTS AS “INFRASTRUCTURE” HERE?
------------------------------------------
IaC is not only about virtual machines. Typical resources declared in code:

  NETWORKING          VPC / VNet, subnets, routes, firewall / security groups,
                      Cloud NAT, VPN / Interconnect, DNS records
  COMPUTE             VMs, instance templates, MIGs / ASGs, GKE / EKS clusters,
                      Cloud Run / Lambda definitions (sometimes)
  STORAGE             Disks, snapshots policies, Cloud Storage / S3 buckets
  DATA                Cloud SQL / RDS instances, backups flags, replicas
  SECURITY / IAM      Service accounts, roles, bindings, KMS keys (carefully)
  LOAD BALANCING      HTTP(S) LB, backend services, health checks, CDN hooks
  OBSERVABILITY       Alert policies, log sinks (advanced but common in orgs)

If you can create it with a cloud API or console, you can usually declare it
in IaC.


1.3  WHY UNIT III MATTERS (STORY ARC OF THE COURSE)
--------------------------------------------------
  Unit 1 — Cloud principles & Well-Architected thinking
            “What does a good cloud design look like?”

  Unit 2 — Disaster Recovery
            “What happens when it fails? How fast can we recover?”

  Unit 3 — Infrastructure as Code   ← YOU ARE HERE
            “How do we create and recreate that design reliably?”

Unit 3 questions in plain language:
  • How do we create cloud resources the same way every time?
  • How do we stop “snowflake” servers that only one engineer understands?
  • How do we rebuild a whole environment after a disaster from code?
  • How do teams change infrastructure safely without breaking production?

Without IaC, even a strong Unit 2 DR plan is slow: someone must remember every
firewall rule, VM size, tag, and IAM binding. With IaC, recovery often becomes:
  restore data from backups  +  re-apply infrastructure from Git.


1.4  THE “SNOWFLAKE SERVER” PROBLEM
-----------------------------------
A snowflake server is a machine that was configured by hand over months —
special packages, one-off fixes, undocumented tweaks. No one can recreate it.

  Real-life parallel:
    Think of a messily customised old PC that “only works for that one lab
    assistant.” If that PC dies, the project dies with it.

IaC + configuration management fight snowflakes by making every server of a
role identical and rebuildable.


1.5  PROBLEMS WITH MANUAL / CONSOLE-ONLY PROVISIONING
-----------------------------------------------------
  PROBLEM                         WHAT GOES WRONG IN REAL LIFE
  ─────────────────────────────────────────────────────────────────────────
  Click-ops (console only)        Steps live in someone’s memory or a
                                  screenshot — not reproducible.

  Environment drift               Dev has different firewall rules than prod.
                                  Bug appears only in production.

  Forgotten resources             Test VMs left running → surprise cloud bill.
                                  Open SSH to 0.0.0.0/0 forgotten after demo.

  Slow disaster recovery          After zone failure, team rebuilds by guessing
                                  from old notes → RTO balloons.

  No peer review                  Junior opens a public database port; nobody
                                  reviewed a PR because there was no PR.

  Scaling pain                    Festival sale needs 40 identical frontends.
                                  Creating them by hand takes hours and errors.

  Knowledge silos                 Only one DevOps person knows “how prod was
                                  built.” They go on leave → risk.


1.6  BENEFITS OF IaC (EXPAND EACH IN 7-MARK ANSWERS)
----------------------------------------------------
  1. Repeatability
     Same definition → same environment shape in another project/region.

  2. Version control
     Git history answers: who changed the load balancer? when? why?

  3. Automation
     CI/CD can run plan/apply; humans review diffs instead of clicking.

  4. Reduced human error
     Fewer missed checkboxes (disk size, tags, backup flags).

  5. Living documentation
     The repository IS the architecture document that stays up to date.

  6. Faster Disaster Recovery (link Unit 2)
     Rebuild networking + compute from code; restore data from backups.

  7. Collaboration
     Pull requests for infrastructure — same culture as application code.

  8. Auditability & compliance
     Auditors can inspect desired state; changes leave a trail.

  9. Cost & scale control
     Instance counts and machine types are parameters you can change and
     review — easier FinOps conversations.


1.7  IMPERATIVE vs DECLARATIVE (HIGH-VALUE EXAM TOPIC)
------------------------------------------------------
  APPROACH      MEANING                         CLOUD EXAMPLE
  ─────────────────────────────────────────────────────────────────────────
  Imperative    You specify HOW — ordered       gcloud compute instances create
                steps / commands                Bash scripts that always “create”

  Declarative   You specify WHAT desired        Terraform resource blocks
                state should be; the tool       CloudFormation templates
                computes create/update/delete   Kubernetes YAML (desired pods)

DETAILED CONTRAST
-----------------
  Imperative script mindset:
    “Create VM A. Then create firewall B. Then create LB C.”
    If you run it twice, you may get duplicate VMs unless you wrote careful
    “if exists, skip” logic yourself.

  Declarative tool mindset:
    “I want exactly 2 healthy frontend VMs behind this LB, with these tags.”
    1st apply → creates missing pieces.
    2nd apply → sees reality already matches → little/no change.
    Change desired count to 5 → tool creates 3 more (not rewrite everything).

EXAM TIP:
  Terraform and CloudFormation are primarily declarative provisioning tools.
  Ansible playbooks are task-oriented (often called procedural) even though
  many modules are written to be idempotent.


1.8  IDEMPOTENCE (MUST-KNOW WORD)
---------------------------------
Idempotent means: applying the same definition multiple times leaves the system
in the same final state — you do not get duplicate chaos.

  Example:
    Desired: “nginx package state = present”
    Run 1: installs nginx
    Run 2: sees nginx already present → does nothing harmful

  Counter-example (non-idempotent shell):
    echo "export PORT=8080" >> ~/.bashrc
    Run 10 times → 10 duplicate lines → messy config

Why exams care:
  IaC and good configuration management aim for safe re-runs — essential for
  automation and for recovering half-finished applies.


1.9  PROVISIONING vs CONFIGURATION (PREVIEW — FULL DETAIL IN §3)
----------------------------------------------------------------
  Infrastructure provisioning
    Create cloud resources via APIs: VM, VPC, disk, LB, managed database.

  Configuration management
    Install and configure software ON servers: packages, app code, users,
    services, config files.

Common industry pattern (remember this sentence):
  “Terraform builds the house; Ansible furnishes the rooms.”
  (CloudFormation can also build the house on AWS.)


1.10  MUTABILITY, DRIFT, AND “SOURCE OF TRUTH”
----------------------------------------------
  Desired state   What Git / IaC says should exist
  Actual state    What currently exists in the cloud project
  Drift           When actual state diverges from desired state
                  (someone clicked in the console and never updated Git)

Best practice: Git is the source of truth. If you must use the console in an
emergency, either:
  • immediately update the IaC to match, or
  • revert the console change and re-apply from code.

Real-life parallel:
  If the architect’s blueprint says 3 floors but workers built 4, the building
  no longer matches the plan. Drift is that mismatch for cloud.


1.11  HOW IaC CONNECTS TO OTHER UNITS & LABS
--------------------------------------------
  • Unit 1 (Well-Architected): encode security/reliability/cost defaults in
    reusable modules so every new app starts “on rails.”
  • Unit 2 (DR): backups restore data; IaC restores the shape of the system —
    together they reduce RTO.
  • Unit 4 (Kubernetes): cluster + node pools often created with Terraform;
    app deployments use YAML (another declarative form).
  • Unit 5 (CI/CD): pipelines plan/apply IaC automatically after review.
  • Practical 7: Automating Infrastructure Deployment with Terraform.

  • Pulumi — IaC in Python / TypeScript / Go / etc.
  • Azure Resource Manager (ARM) / Bicep — Azure-native IaC


2.2  TERRAFORM (HashiCorp) — DEEP DIVE
--------------------------------------

WHAT IT IS
  Terraform is a widely used IaC tool that declares infrastructure in HCL
  (HashiCorp Configuration Language) and uses provider plugins to talk to
  cloud APIs (google, aws, azurerm, kubernetes, …).

WHY INDUSTRY LOVES IT
  • One language / workflow across clouds and SaaS
  • Strong plan/apply preview culture
  • Huge module ecosystem
  • Fits Git + PR + CI very well

CORE BUILDING BLOCKS
  Provider     Plugin that knows how to call a cloud API
  Resource     Something you want to create/manage
               (e.g. google_compute_instance)
  Data source  Read existing information without creating
  Variable     Input parameter (region, project, count)
  Output       Value exported after apply (LB IP)
  Module       Reusable package of resources
  State        Record of what Terraform manages in the real world

ESSENTIAL COMMANDS (KNOW WHAT EACH DOES)
  terraform init      Download providers/modules; prepare backend
  terraform fmt       Auto-format HCL
  terraform validate  Syntax / basic consistency check
  terraform plan      Preview create / change / destroy (NO changes yet)
  terraform apply     Execute the plan
  terraform destroy   Remove resources tracked in state
  terraform state …   Advanced inspection / surgery of state
  terraform output    Print declared outputs

LIFE CYCLE OF A CHANGE
  Edit .tf files → fmt/validate → plan → human reviews plan → apply → verify

TERRAFORM STATE — WHY IT MATTERS
  State maps your resource blocks to real cloud IDs. Without state, Terraform
  cannot know “this google_compute_instance.web is instance ID xyz.”

  Local state (terraform.tfstate on laptop):
    OK for personal labs; risky for teams.

  Remote state (e.g. GCS / S3 backend) + locking:
    Team-safe; prevents two applies at once; state survives laptop loss.

  Security note:
    State may contain sensitive values — restrict who can read the bucket.

CONCEPTUAL EXAMPLE 1 — GCP FIREWALL (lab style)
-----------------------------------------------
  provider "google" {
    project = var.project_id
    region  = var.region
  }

  resource "google_compute_firewall" "fw_fe" {
    name    = "fw-fe"
    network = "default"

    allow {
      protocol = "tcp"
      ports    = ["8080"]
    }

    target_tags   = ["frontend"]
    source_ranges = ["0.0.0.0/0"]  # tighten for real production!
  }

CONCEPTUAL EXAMPLE 2 — SIMPLE VM SHAPE
--------------------------------------
  resource "google_compute_instance" "web" {
    name         = "web-1"
    machine_type = var.machine_type
    zone         = var.zone

    boot_disk {
      initialize_params {
        image = "debian-cloud/debian-12"
      }
    }

    network_interface {
      network = "default"
      access_config {}   # ephemeral public IP — avoid if private-only needed
    }

    tags = ["frontend"]
  }

LAB → TERRAFORM RESOURCE MAP
----------------------------
  Manual / gcloud action                 Typical Terraform resource
  ─────────────────────────────────────────────────────────────────────────
  Create Compute Engine VM               google_compute_instance
  Firewall rule                          google_compute_firewall
  Instance template                      google_compute_instance_template
  Managed Instance Group                 google_compute_instance_group_manager
  Autoscaler                             google_compute_autoscaler
  GKE cluster                            google_container_cluster
  Cloud Storage bucket                   google_storage_bucket
  Cloud SQL instance                     google_sql_database_instance

REAL-LIFE USE
  A fintech or e-commerce platform team keeps “network + GKE + IAM baseline”
  as Terraform modules. New microservice teams consume the module instead of
  inventing a new VPC every time.


2.3  AWS CLOUDFORMATION — DEEP DIVE
----------------------------------

WHAT IT IS
  CloudFormation is AWS’s native IaC service. You author a template (YAML or
  JSON) declaring AWS resources; AWS creates and updates them as a stack.

KEY TERMS
  Template     The YAML/JSON definition file
  Stack        A deployed group of resources managed together
  Change set   Preview of what an update will do (similar spirit to plan)
  Nested stack Stack that calls other templates (modularity)

STRENGTHS
  • Deepest AWS feature coverage; first-party support
  • AWS manages reconciliation; you don’t maintain a separate state file
  • Integrates with AWS IAM, CloudTrail, and organisation policies

LIMITS (use in comparison answers)
  • AWS-only — not ideal if you also run GCP/Azure in one toolchain
  • Large templates become hard to read without nesting / modules
  • Multi-cloud companies often standardise on Terraform instead

CONCEPTUAL YAML SNIPPET — S3 BUCKET
-----------------------------------
  AWSTemplateFormatVersion: "2010-09-09"
  Description: CC-II demo bucket
  Resources:
    WebBucket:
      Type: AWS::S3::Bucket
      Properties:
        BucketName: my-cc2-demo-bucket
        VersioningConfiguration:
          Status: Enabled

CONCEPTUAL IDEA — VPC + EC2 (what a larger template contains)
------------------------------------------------------------
  Resources might include AWS::EC2::VPC, AWS::EC2::Subnet,
  AWS::EC2::SecurityGroup, AWS::EC2::Instance, AWS::ElasticLoadBalancingV2::...
  Updating the stack applies a change set after review.

REAL-LIFE USE
  An enterprise that is fully on AWS may mandate CloudFormation (or CDK that
  synthesises CloudFormation) so every account’s networking follows a central
  approved template — auditors love that consistency.


2.4  ANSIBLE — DEEP DIVE
------------------------

WHAT IT IS
  Ansible is an automation tool used mainly for configuration management and
  application deployment. You write YAML playbooks that list tasks. In the
  common model, a control node pushes configuration to targets over SSH —
  agentless (no long-running agent required on each server).

WHERE IT FITS RELATIVE TO TERRAFORM / CLOUDFORMATION
  Terraform / CloudFormation  → create the VM, disk, network, LB
  Ansible                     → install packages, copy app, write configs,
                                start services, create users

  They complement; they are not pure rivals.

ANSIBLE VOCABULARY
  Inventory     List of hosts / groups (web, db, …)
  Playbook      YAML file with plays and tasks
  Module        Built-in action (apt, copy, service, user, …)
  Role          Reusable bundle of tasks/templates/files
  Handler       Task that runs on notification (e.g. restart nginx once)

CONCEPTUAL PLAYBOOK (SERVER SETUP)
----------------------------------
  - name: Configure Fancy Store frontend
    hosts: frontend
    become: true
    tasks:
      - name: Install Node.js and npm
        apt:
          name: ["nodejs", "npm"]
          state: present
          update_cache: true

      - name: Ensure app user exists
        user:
          name: nodeapp
          state: present

      - name: Deploy application files
        copy:
          src: ./app/
          dest: /opt/fancystore/
          owner: nodeapp

      - name: Ensure app service is running
        service:
          name: fancystore
          state: started
          enabled: true

ANSIBLE vs STARTUP SCRIPT (SAME IDEA, DIFFERENT FORM)
-----------------------------------------------------
Compute Engine startup scripts that:
  apt-get install … → install Node → gsutil cp app → configure Supervisor
are configuration management expressed as Bash.

Ansible expresses the same intent as readable, reusable, idempotent tasks —
better for fleets of servers and team collaboration.

REAL-LIFE USE
  After Terraform creates 20 identical VMs for a food-delivery API, Ansible
  (or an image baked by Packer + Ansible) ensures every VM has the same app
  version, nginx config, and monitoring agent — no SSH snowflakes.


2.5  HOW TO CHOOSE (DECISION GUIDE)
-----------------------------------
  SITUATION                                   PREFER
  ─────────────────────────────────────────────────────────────────────────
  Multi-cloud or may switch clouds            Terraform
  AWS-only org with AWS-native standards      CloudFormation (or CDK→CFN)
  Configure OS/app on existing servers        Ansible
  GCP student lab / Practical 7               Terraform
  Need both create + configure                Terraform + Ansible (together)
  Kubernetes app deploy only                  kubectl/Helm YAML (Unit 4)
                                              — still “declarative code”


2.6  FULL SIDE-BY-SIDE FOR LONG ANSWERS
---------------------------------------
  Aspect           Terraform            CloudFormation         Ansible
  ─────────────────────────────────────────────────────────────────────────
  Primary job      Provision infra      Provision AWS infra    Configure hosts
  Language         HCL                  JSON / YAML            YAML playbooks
  Cloud scope      Multi-cloud          AWS only               Multi / on-prem

                                                               for cloud objects

2.7  WORKED MINI-ANSWER — “GIVE ONE CONCEPTUAL EXAMPLE”
-------------------------------------------------------
Question style: Write a short conceptual example showing how any one tool can
create a cloud resource.

Model (Terraform):
  “Using Terraform, we declare a google_compute_firewall resource in HCL with
   name, network, allowed tcp ports, and target tags. After terraform plan and
   terraform apply, GCP creates that firewall rule via API. A second apply makes

  “An Ansible playbook task uses the apt module with state=present to ensure
   nginx is installed on all hosts in the web inventory group.”

3. CONFIGURATION MANAGEMENT & AUTOMATION
================================================================================

3.1  DEFINITION
---------------
Configuration Management (CM) is the discipline of maintaining a system’s
software and settings in a known, desired state — which packages are installed,
which files exist with which content, which services are running, which users
exist — typically through automation rather than manual SSH.

Goal in one line:
  “Every server of the same role looks the same — and stays that way.”


3.2  PROVISIONING vs CONFIGURATION (EXAM GOLD)
----------------------------------------------
  DIMENSION         INFRASTRUCTURE              SOFTWARE CONFIGURATION
                    PROVISIONING                (CM)
  ─────────────────────────────────────────────────────────────────────────
  Core question     What cloud resources        What is installed and
                    exist?                      configured on the machine?
  Examples          VPC, subnet, VM, disk,      apt/yum packages, runtimes,
                    firewall, LB, Cloud SQL     app binaries, nginx.conf,
                    instance                    systemd units, users, cron
  Typical tools     Terraform, CloudFormation   Ansible, Chef, Puppet,
                                                Salt, cloud-init / startup
                                                scripts
  Timing            Create/update via cloud     After the OS/VM exists
                    control-plane APIs          (SSH, agent, or metadata)
  Failure mode      Wrong network design,       Drifted packages, missing
                    missing HA                  service, wrong config file

Memory hook:
  Blank VM created = provisioning done.
  Fancy Store Node app running on it = configuration done.


3.3  WHY AUTOMATION BEATS MANUAL SSH
------------------------------------
Manual configuration causes:
  • Configuration drift — Friday hotfix on server-3 never copied to server-1/2
  • Slow scale-out — cannot confidently add 25 nodes before a sale
  • Fragile DR — rebuild steps live in chat history
  • Security gaps — one host missing the hardening checklist

Automation means: empty machine → desired role in minutes, same every time.


3.4  CONFIGURATION MANAGEMENT TOOLS (LANDSCAPE)
-----------------------------------------------
  TOOL       TYPICAL MODEL                 NOTES FOR CC-II
  ─────────────────────────────────────────────────────────────────────────
  Ansible    Push over SSH; agentless      Syllabus focus; YAML playbooks
  Chef       Recipes; client often pulls   Enterprise history; Ruby DSL
  Puppet     Declarative manifests         Classic agent-based CM
  Salt       Remote execution / states     Strong at large fleets
  cloud-init Runs at first boot from       Related idea on cloud VMs
             metadata

For written exams, emphasise Ansible; mention others briefly for viva.


3.5  PUSH vs PULL MODELS
------------------------
  Push model (common Ansible):
    Control node connects to targets and pushes desired config.

  Pull model (common in some Chef/Puppet setups):
    Each node periodically pulls its desired config from a central server.

Exam sufficiency:
  “Ansible typically uses a push model over SSH and is agentless.”


3.6  TYPICAL AUTOMATED SERVER SETUP STEPS (MEMORISE SEQUENCE)
------------------------------------------------------------
Use this 8-step ladder in Q.11-style answers:

  1. Update package index / install base OS packages
  2. Install application runtime (Node.js, Java, Python, …)
  3. Create a dedicated non-root application user
  4. Pull application code or build artifacts
     (Git, Cloud Storage, Artifact Registry)
  5. Write configuration (ports, env files; secrets from Secret Manager)
  6. Configure a process manager (systemd / Supervisor)
  7. Start and enable the service on boot
  8. Verify health (curl localhost:PORT/health, check logs)

FANCY STORE / LAB PARALLEL
--------------------------
  apt-get update & install tools
    → install Node.js
    → gsutil cp application from bucket
    → configure Supervisor to keep process alive
    → run as user `nodeapp` (not root)

That Bash startup script IS a configuration-management procedure.


3.7  IDEMPOTENT CONFIGURATION TASKS
-----------------------------------
Good CM says:
  • Package state=present (not “always reinstall”)
  • Service state=started (not “restart forever in a loop”)
  • File content must equal X (not “append X every run”)

Bad CM says:
  • Unconditional echo >> file
  • wget installer every time without checks
  • Start service without checking if already healthy

Idempotence lets you re-run the playbook safely after partial failure.


3.8  IMAGES vs BOOTSTRAP (TWO STRATEGIES)
-----------------------------------------
  Strategy A — Bootstrap at boot
    Start from a generic OS image; startup script / Ansible configures on boot.
    Flexible; slower cold start; more moving parts at runtime.

  Strategy B — Golden / baked image
    Use Packer (or similar) + Ansible to bake packages into a custom image;
    VMs boot almost ready. Faster scale-out; image versioning required.

Real-life:
  Food-delivery or UPI API fleets often prefer baked images or containers so
  new capacity comes online in seconds during traffic spikes.


3.9  CONFIGURATION DRIFT — DETECTION & RESPONSE
-----------------------------------------------
Drift example:
  On-call engineer SSHes into prod and edits nginx.conf by hand during an
  incident — then forgets to update the playbook.

Response:
  • Re-run Ansible to converge back to Git’s desired state, OR
  • Update the playbook/image to include the legitimate fix, then roll out

Link to IaC philosophy: desired state lives in version control.

4. WRITING & MANAGING INFRASTRUCTURE AS CODE
================================================================================

4.1  THE STANDARD WORKFLOW (MEMORISE EXACTLY)
---------------------------------------------
  Write → Version Control → Review → Plan → Apply → Verify

  STEP               WHAT YOU DO                          WHY IT MATTERS
  ─────────────────────────────────────────────────────────────────────────
  Write              Author .tf / CFN templates /         Capture intent as code
                     playbooks
  Version Control    Commit on a feature branch in Git    History + collaboration
  Review             Pull request / peer review           Catch dangerous opens,
                                                          cost bombs, deletes
  Plan               terraform plan / CFN change set      Preview creates/changes/
                                                          destroys BEFORE impact
  Apply              terraform apply / stack update       Execute approved change
  Verify             Health checks, smoke tests,          Confirm reality matches
                     dashboards, IAM spot-checks          intent

WHY “PLAN BEFORE APPLY” IS A 7-MARK FAVOURITE
---------------------------------------------
Plan is a safety net. It answers:
  • Will this destroy a database?
  • Will this replace a VM (downtime)?
  • Did I accidentally remove a firewall rule?

Real-life scare story (exam-friendly):
  An engineer renames a Terraform resource block without a state move. Plan
  shows: destroy old DB + create new empty DB. Without reading the plan, apply
  would be catastrophic. Plan-before-apply saves production.


4.2  TYPICAL TERRAFORM PROJECT LAYOUT
-------------------------------------
  my-infra/
    versions.tf / providers.tf   required_version + provider blocks
    main.tf                      primary resources (or split by concern)
    network.tf                   VPC / firewall (optional split)
    variables.tf                 input variable declarations
    outputs.tf                   exported values (LB IP, instance names)
    terraform.tfvars             env-specific values (careful with secrets)
    modules/
      network/                   reusable VPC module
      compute/                   reusable MIG/VM module
    README.md                    how to init/plan/apply

Modules = write once, reuse for many applications / environments.


4.3  VARIABLES, OUTPUTS, WORKSPACES, ENV FILES
----------------------------------------------
  Variables     Parameterise project_id, region, machine_type, instance_count
  Outputs       Publish load_balancer_ip for app teams / DNS
  tfvars        dev.tfvars vs prod.tfvars (different sizes/regions)
  Workspaces    Alternate approach to separate env state (know the idea)

Rule:
  Same code shape for all environments; different inputs for size and names.
  Never put production passwords in committed tfvars.


4.4  REMOTE STATE, LOCKING, AND TEAM SAFETY
-------------------------------------------
Problems with laptop-only state:
  • Lost laptop → lost mapping of resources
  • Two engineers apply at once → race / corruption risk
  • No shared visibility

Better:
  • Remote backend (GCS bucket / S3) for state
  • State locking enabled
  • Tight IAM on who can read/write state
  • Optional: separate state per environment / per component


4.5  SECRETS MANAGEMENT (SECURITY — ALWAYS MENTION)
---------------------------------------------------
NEVER commit:
  • API keys, DB passwords, private keys
  • Service account JSON key files
  • Long-lived access tokens in plaintext

PREFER:
  • GCP Secret Manager / AWS Secrets Manager / SSM Parameter Store
  • Short-lived identities (Workload Identity / instance SA)
  • CI secrets injected at pipeline runtime
  • Encryption + least-privilege IAM

IaC may create the secret resource or reference it — but secret values should
not live in Git history.


4.6  CODE REVIEW CHECKLIST FOR INFRASTRUCTURE PRs
-------------------------------------------------
Reviewers should ask:
  [ ] Are firewall / security group rules least-privilege?
  [ ] Any 0.0.0.0/0 on SSH/RDP?
  [ ] Public IP required, or private + IAP / bastion better?
  [ ] Database deletion protection / backups enabled?
  [ ] Machine sizes / quotas reasonable (cost)?
  [ ] Tags/labels present (owner, env, cost-centre)?
  [ ] Plan reviewed for unexpected destroys?
  [ ] Secrets absent from the diff?


4.7  CHANGE MANAGEMENT FOR STATEFUL vs STATELESS
------------------------------------------------
  Stateless compute (web frontends behind LB)
    Easier: rolling replace, new template version, scale out/in.

  Stateful data (databases, disks with unique data)
    Dangerous: replacement can mean data loss if misunderstood.
    Prefer snapshots/PITR (Unit 2), blue/green logical migration, careful plans.

Exam line:
  “Treat database changes in IaC with extra caution; always read the plan.”


4.8  LINK TO CI/CD (PREVIEW OF UNIT 5)
--------------------------------------
  Developer opens PR with Terraform changes
    → CI: fmt, validate, plan (comment plan on PR)
    → Human review + approval
    → CD: apply to staging automatically
    → Production apply may require manual approval gate

Tools you will see later: Cloud Build, GitHub Actions, GitLab CI, Jenkins.


4.9  ROLLBACK THINKING
----------------------
Rollback is not always “terraform apply old commit” if data already migrated.
Still, Git tags for infrastructure releases let you know which commit matched
which production behaviour — essential for incident response.

5. BEST PRACTICES FOR VERSIONING & SCALING INFRASTRUCTURE
================================================================================

5.1  VERSIONING PRACTICES
-------------------------
  PRACTICE                            WHY
  ─────────────────────────────────────────────────────────────────────────
  Keep all IaC in Git                 Single source of truth
  Small PRs with clear messages       Reviewable intent
  Tag releases (v1.0.0, v1.1.0)       Correlate app release ↔ infra release
  Pin provider / module versions      Avoid surprise breakage on init
  Prefer immutable artifacts          New image/template version > mutate old
  Document breaking changes           Team knows migration steps

Immutable infrastructure mindset:
  Do not SSH into production and “fix forever.”
  Build a new version → deploy → retire the old version.


5.2  SCALING WITH IaC (NOT COPY-PASTE)
--------------------------------------
Make capacity a parameter:

  TECHNIQUE                     EXAMPLE
  ─────────────────────────────────────────────────────────────────────────
  Parameterise counts           var.instance_count / MIG target_size
  Encode autoscaling            CPU target 60% → add/remove VMs automatically
  Horizontal scaling            More identical instances behind LB
  Vertical scaling (careful)    Larger machine_type via variable
  Multi-zone flags              Spread for HA (Unit 2 link)
  Same module, different env    Dev size=1, Prod size=6 via tfvars

PEAK EVENT EXAMPLE (CAMPUS / COMMERCE)
-------------------------------------
  Normal week:   MIG size = 2
  Result day / sale: MIG size = 12 (same Terraform, new tfvars or variable)
  After peak:    scale down → control cost (FinOps)

Without IaC, that change is a panicked console marathon.


5.3  ENVIRONMENT PROMOTION PATH
-------------------------------
  Dev  →  Staging  →  Production

Rules of thumb:
  • Promote the same module version forward
  • Change inputs (size, domain, replica count), not random forks of code
  • Break things in Dev/Staging before Production apply
  • Production may require extra approvals


5.4  TESTING & POLICY GUARDS
----------------------------
  • terraform fmt / validate in CI (mandatory baseline)
  • Plan output attached to PR
  • Policy as Code examples: deny open SSH; require labels; deny huge VMs
  • Smoke tests after apply: hit /health, check LB backends healthy
  • Periodic drift detection jobs (compare actual vs desired)


5.5  OPERATIONAL CHECKLIST (PRINTABLE)
--------------------------------------
  [ ] Git is source of truth — minimise long-lived console drift
  [ ] Always plan before apply in shared environments
  [ ] Remote state + locking for teams
  [ ] No secrets in Git
  [ ] Modules for reuse; variables for differences
  [ ] Labels/tags for ownership and cost
  [ ] Deletion protection + backups for data stores
  [ ] DR runbook includes “re-apply IaC” step (Unit 2)
  [ ] Destroy understood before anyone runs it


5.6  ANTI-PATTERNS (WRITE AS “AVOID”)
------------------------------------
  • Click in console and never update code → permanent drift
  • One gigantic root module nobody understands
  • “God” credentials in plaintext committed tfvars
  • terraform apply to prod from a laptop with no plan review
  • IaC used only on day 1, then forever SSH snowflake management
  • Copy-pasting 50 nearly identical resource blocks instead of count/modules

6. EXAM QUICK-REVISION (IMPORTANT QUESTIONS + MODEL ANSWERS)

--------------------------------------------------------------------------------
Q1 / MiD1 Q.9 style — What is IaC? Why preferred over console? Benefits + scenario
--------------------------------------------------------------------------------
MODEL ANSWER (≈7 marks structure):

Definition:
  Infrastructure as Code means provisioning and managing infrastructure using
  machine-readable definition files stored in version control, instead of manual
  console clicking or one-off commands.

Why preferred over manual console provisioning:
  Console work is hard to reproduce, poorly documented, difficult to review,
  slow for disaster recovery, and causes environment drift between
  Dev/Staging/Prod. IaC makes infrastructure repeatable, reviewable, and
  automatable.

Benefits (write 5–6 briefly):
  Repeatability; Git history; automation via CI/CD; fewer human errors;
  living documentation; faster rebuild for DR; collaboration via pull requests;
  auditability.

Practical scenario:
  A university result portal needs matching Dev/Test/Prod and must scale out on
  result day. With Terraform, one module defines VPC, MIG, load balancer, and
  database settings; only variables change per environment. Before result day,
  engineers apply a larger instance_count from the same code. After a zone
  issue, they rebuild from Git and restore data from backups (Unit 2).


--------------------------------------------------------------------------------
Q2 / MiD1 Q.10 style — Compare Terraform, CloudFormation, Ansible + example
--------------------------------------------------------------------------------
MODEL ANSWER:

  Tool              Purpose              Language     Cloud        Approach
  Terraform         Provision infra      HCL          Multi-cloud  Declarative
  CloudFormation    Provision AWS        JSON/YAML    AWS only     Declarative
  Ansible           Configure servers    YAML         Multi/onprem Task/procedural
                    (CM focus)           playbooks                 (idempotent
                                                                   modules)

Discussion:
  Terraform and CloudFormation both declare desired cloud resources.
  Terraform is multi-cloud; CloudFormation is AWS-native with stacks/change sets.
  Ansible focuses on software configuration (packages, files, services) usually
  pushed over SSH; often used after VMs are provisioned.

Conceptual example (pick one):
  Terraform google_compute_firewall / google_compute_instance block applied with
  plan/apply; OR CloudFormation AWS::S3::Bucket in a stack; OR Ansible apt module
  ensuring nginx is present.


--------------------------------------------------------------------------------
Q3 / MiD1 Q.11 style — Configuration management vs provisioning + Ansible steps
--------------------------------------------------------------------------------
MODEL ANSWER:

Configuration Management:
  Keeping software and settings on servers in a desired state through automation.

Difference:
  Provisioning creates cloud resources (VM, VPC, LB, managed DB).
  Configuration installs/configures software on those resources (runtime, app,
  service manager, users).

Ansible role:
  Agentless automation using YAML playbooks; typically push over SSH; modules
  ensure packages/services/files match desired state idempotently.

Typical automated server setup steps:
  1) update/install base packages  2) install runtime  3) create app user
  4) pull artifacts  5) write config/secrets refs  6) configure systemd/Supervisor
  7) start/enable service  8) health-check verification

Lab parallel: Compute Engine startup script for Fancy Store is the same pattern.


--------------------------------------------------------------------------------
Q4 / MiD1 Q.12 style — Workflow + versioning/scaling + plan-before-apply
--------------------------------------------------------------------------------
MODEL ANSWER:

Workflow:
  Write definitions → commit to Version Control (Git) → Review via PR →
  Plan (preview) → Apply → Verify with health checks/smoke tests.

Plan before Apply:
  Shows creates/updates/destroys before any irreversible change. Prevents
  accidental database destruction, unexpected replacements, and costly mistakes.

Versioning best practices:
  Git as source of truth; tagged releases; pin provider/module versions;
  immutable images/templates; no secrets in Git; remote state + locking.

Scaling best practices:
  Parameterise instance counts; encode autoscaling; reuse modules across envs;
  promote Dev→Staging→Prod; scale up for peaks (result day/sale) and down after
  for cost control.


--------------------------------------------------------------------------------
ADDITIONAL SHORT Q&A
--------------------------------------------------------------------------------
Q5. Imperative vs declarative?
A5. Imperative = HOW steps (gcloud create…). Declarative = WHAT desired state
    (Terraform/CloudFormation); tool converges.

Q6. What is idempotence?
A6. Re-applying the same definition yields the same end state without duplicate
    harmful side effects.

Q7. What is drift?
A7. Actual cloud state differs from desired state in Git (often from console
    changes). Fix by re-applying code or updating code to match intentionally.

Q8. How does IaC help DR (Unit 2)?
A8. Backups restore data; IaC restores infrastructure shape; together reduce RTO
    vs manual rebuild from memory.

Q9. Why remote state?
A9. Team sharing, durability, locking against concurrent applies, better security
    controls than laptop files.

Q10. Snowflake server?
A10. Uniquely hand-configured machine that cannot be reliably recreated — IaC/CM
     are the antidote.

8. REAL-LIFE CASE STUDIES (QUOTE THESE IN LONG ANSWERS)
================================================================================

--------------------------------------------------------------------------------
CASE 1 — Campus ERP / Exam Result Portal (Parul-style scenario)
--------------------------------------------------------------------------------
Problem:
  Most of the year traffic is moderate. On result day, thousands of students
  hit the portal. Historically, IT staff created extra VMs by hand and forgot
  some firewall rules — outages and uneven configs.

IaC approach:
  Terraform module for VPC + MIG + HTTPS load balancer + Cloud SQL flags.
  Dev/Staging/Prod share the module; prod.tfvars sets higher instance_count
  only for the result window. Ansible/startup script ensures identical app
  install on every VM.

Teaching point:
  Scale becomes a variable change + plan/apply — not a panic click-athon.
  After the event, scale down for cost.


--------------------------------------------------------------------------------
CASE 2 — Food Delivery App During Dinner Peak
--------------------------------------------------------------------------------
Problem:
  Order API must add many identical frontend/order services between 7–10 PM.
  Manual images drift; one host misses a security patch.

IaC + CM approach:
  Terraform manages MIG autoscaling policy and LB.
  Golden image baked with Ansible (or containers later in Unit 4).
  New capacity boots ready to serve.

Teaching point:
  Provisioning (capacity) + configuration (identical software) both required.


--------------------------------------------------------------------------------
CASE 3 — UPI / Payments Platform (Low RTO mindset)
--------------------------------------------------------------------------------
Problem:
  Minutes of downtime mean failed payments and lost trust. Rebuild from memory
  is unacceptable (Unit 2 RTO pressure).

IaC approach:
  Entire networking, GKE/MIG baselines, IAM, and health checks defined in Git.
  DR runbook: restore data stores from PITR/backups + re-apply Terraform in
  another zone/region + traffic shift.

Teaching point:
  IaC is not only “DevOps convenience” — it is a Disaster Recovery accelerator.


--------------------------------------------------------------------------------
CASE 4 — E-commerce Festival Sale (Flipkart/Amazon-style thinking)
--------------------------------------------------------------------------------
Problem:
  Traffic multiplies; teams must clone production-like staging to test sale
  configs, then push capacity changes quickly.

IaC approach:
  Same Terraform codebase; staging uses smaller counts; production scales out
  via variables/autoscaling. PR review catches an open security group before
  the sale.

Teaching point:
  Review + plan prevent festival-day security and cost disasters.


--------------------------------------------------------------------------------
CASE 5 — Startup “Works on My Cloud Project” Drift
--------------------------------------------------------------------------------
Problem:
  Founder built prod in console. New engineer cannot recreate Dev. Onboarding
  takes weeks; every fix is unique.

IaC approach:
  Export/rewrite architecture into Terraform over a sprint. Enforce “no console
  changes without updating Git.” New environments appear in hours.

Teaching point:
  IaC converts tribal knowledge into an onboarding asset.


--------------------------------------------------------------------------------
CASE 6 — Accidental Database Destroy Caught by Plan
--------------------------------------------------------------------------------
Problem:
  Rename of a resource block would destroy and recreate a database.

IaC workflow save:
  terraform plan shows destroy/create. Reviewer blocks merge. Team uses state
  move / lifecycle safeguards instead.

Teaching point:
  Plan-before-apply is not bureaucracy — it is production safety.


--------------------------------------------------------------------------------
CASE 7 — Hybrid: Terraform Creates VMs, Ansible Configures App
--------------------------------------------------------------------------------
Problem:
  Cloud team owns accounts/network; app team owns runtime and deploy.

Split:
  Terraform (provisioning) for VPC/VM/LB.
  Ansible (configuration) for app install and service config.
  Clear ownership; fewer snowflakes.

Teaching point:
  Syllabus tools are complementary when used at the right layer.

ONE-LINE DEFINITIONS
  IaC            Infra defined/managed as versioned code
  Declarative    Declare desired state; tool converges
  Imperative     Specify step-by-step commands
  Idempotent     Safe to re-apply; same end state
  Drift          Actual ≠ desired (often console edits)
  Provisioning   Create cloud resources
  CM             Keep software/settings desired on hosts
  State (TF)     Mapping of code resources ↔ real IDs
  Plan           Preview before apply
  Snowflake      Unique unreproducible hand-built server

TOOL CHEAT
  Terraform         Multi-cloud provision, HCL, state, plan/apply
  CloudFormation    AWS-only provision, YAML/JSON, stacks/change sets
  Ansible           CM playbooks, YAML, SSH push, agentless

  Write → Git → Review → Plan → Apply → Verify
