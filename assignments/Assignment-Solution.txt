Q.1  Fundamental principles of cloud computing                         [7]
--------------------------------------------------------------------------------

QUESTION RECALL
Explain On-demand Self-service, Broad Network Access, Resource Pooling,
Rapid Elasticity and Measured Service. Discuss how they give higher
throughput and better cost control than traditional on-premises computing.

MODEL ANSWER

Introduction
------------
Cloud computing is a model for delivering IT resources (compute, storage,
network, platforms, software) over the network on a pay-as-you-go basis.
The National Institute of Standards and Technology (NIST) describes five
fundamental characteristics / principles. These principles explain WHY cloud
is different from buying and running your own data-centre hardware.

1. On-demand Self-service
-------------------------
Users can provision computing capabilities (VMs, storage buckets, databases)
automatically through a web console or API, without human interaction with
each service provider staff member for every request.

  Traditional on-premises: raise a purchase order → wait weeks for servers →
  rack, cable, install OS.
  Cloud: click Create VM / run gcloud → ready in minutes.

  Exam value: reduces lead time; developers and ops teams move faster.

2. Broad Network Access
-----------------------
Capabilities are available over the network and accessed through standard
mechanisms (HTTPS, APIs) from heterogeneous client platforms — laptops,
mobiles, tablets, other servers, partner systems.

  This enables remote work, multi-campus access, and integration with
  mobile apps. Private connectivity (VPN / Interconnect) can also be used
  for hybrid designs while still following the “network-accessible” idea.

3. Resource Pooling
-------------------
The provider’s computing resources are pooled to serve multiple customers
using a multi-tenant model. Different physical and virtual resources are
dynamically assigned according to demand. The customer generally has no
control or knowledge of the exact physical server, but may specify location
at a higher level (region / zone).

  Benefits: better utilisation of hardware, economies of scale for the
  provider, and capacity available to customers without owning spare racks.

4. Rapid Elasticity
-------------------
Resources can be elastically provisioned and released to scale outward and
inward with demand. To the consumer, capabilities often appear unlimited
and can be purchased in any quantity at any time.

  Cloud mechanisms: Managed Instance Groups, Kubernetes HPA, serverless
  functions, autoscaling databases.
  Result: handle result-day or festival sale spikes without permanent
  over-purchase of hardware.

5. Measured Service
-------------------
Cloud systems automatically control and optimise resource use by metering
(CPU-hours, GB-month, network egress, API requests). Usage can be monitored,
controlled, and reported — enabling transparency for both provider and
customer (billing / FinOps).

  This supports chargeback, budgets, and “pay for what you use” instead of
  paying for idle capital equipment year-round.

How these principles improve THROUGHPUT vs on-premises
------------------------------------------------------
  Throughput = how much useful work the system can complete per unit time
  (requests/sec, students served, transactions processed).

  • Elasticity + self-service → add capacity in minutes when load rises,
    so queues do not grow and response time stays acceptable.
  • Broad network access + global regions → place apps closer to users,
    reducing latency and increasing completed transactions.
  • Pooling → provider always has spare capacity you can borrow; you are
    not limited by last year’s hardware purchase.

How these principles improve COST CONTROL vs on-premises
--------------------------------------------------------
  • Measured service → see exactly what costs money; delete idle resources.
  • Elasticity → scale IN after the peak; stop paying for unused VMs.
  • Pooling / OpEx model → avoid large CapEx for servers that sit idle
    most of the academic year.
  • On-premises often forces over-provisioning “for the worst day,” so
    average utilisation (and value for money) stays low.

Detailed real-life example (write this)
---------------------------------------
A university result portal:
  • Normal days: 2 small web instances (low cost — measured + scaled in).
  • Result day: autoscaling adds instances (elasticity + self-service APIs).
  • Students access from phones/laptops across cities (broad network access).
  • After a week: scale back down — bill drops (cost control).
  On-premises would need enough servers for peak all year → poor cost
  efficiency and still risk of overload if peak exceeds purchase.

Diagram (draw in booklet)
-------------------------
  On-prem:  [Fixed 20 servers] ----always on----> high idle cost
  Cloud:    [2 servers] --peak--> [20 servers] --quiet--> [2 servers]
            (metered bill follows the curve)

Conclusion
----------
The five principles — on-demand self-service, broad network access, resource
pooling, rapid elasticity, and measured service — work together so
organisations can deliver higher throughput during demand spikes while
paying mainly for used capacity, which traditional fixed on-premises
capacity cannot match as flexibly.

Q.2  Shared Responsibility Model                                       [7]
--------------------------------------------------------------------------------

QUESTION RECALL
Define Shared Responsibility Model. Compare provider vs customer for IaaS,
PaaS, SaaS with examples. Explain one security risk if misunderstood.

MODEL ANSWER

Introduction / Definition
-------------------------
The Shared Responsibility Model is a security and operations framework that
clearly divides duties between the Cloud Service Provider (CSP) and the
customer. It corrects a common myth: “If we move to cloud, the provider is
responsible for everything.”

  In one exam line:
  “Security OF the cloud is mainly the provider’s job; security IN the cloud
   is mainly the customer’s job — and the exact split depends on IaaS/PaaS/SaaS.”

Why the model exists
--------------------
Cloud stacks have many layers: physical building, network, hypervisor, OS,
runtime, application, data, identity. Someone must patch, configure, and
monitor each layer. The model assigns each layer so nothing is “assumed”
to be covered.

Detailed comparison table (memorise and draw)
---------------------------------------------
  Layer / Duty              IaaS (GCE VM)     PaaS (App Engine)   SaaS (Gmail)
  ────────────────────────────────────────────────────────────────────────────
  Physical data centre        Provider          Provider            Provider
  Hardware / power / cooling  Provider          Provider            Provider
  Hypervisor / host network   Provider          Provider            Provider
  Guest OS patching           CUSTOMER          Provider            Provider
  Runtime / middleware        CUSTOMER          Provider            Provider
  Application code            CUSTOMER          CUSTOMER            Provider
  Data classification         CUSTOMER          CUSTOMER            CUSTOMER
  Identity & access (IAM)     CUSTOMER          CUSTOMER            CUSTOMER*
  Network config (VPC/fw)     CUSTOMER          Limited / shared    Provider
  Endpoint device security    CUSTOMER          CUSTOMER            CUSTOMER

  * Customer configures accounts, MFA, sharing policies even in SaaS.

Suitable examples (expand each)
-------------------------------
IaaS — Google Compute Engine VM hosting a Node.js app
  Provider: secures building, hardware, hypervisor.
  Customer: chooses OS image, applies security updates, configures firewall
  tags, installs Node, hardens SSH, manages app secrets, encrypts disks if
  required, monitors logs.
  If the customer leaves port 22 open to 0.0.0.0/0 with a weak password,
  that breach is a CUSTOMER failure, not “Google was hacked.”

PaaS — Google App Engine
  Provider: manages runtime, scaling, OS patching under the platform.
  Customer: writes secure application code, manages data access, sets IAM
  for who can deploy, protects API keys, validates input (SQL injection etc.).
  Customer does NOT patch the guest OS — that duty moved to the provider.

SaaS — Google Workspace / Gmail
  Provider: runs the entire application stack.
  Customer: still responsible for user lifecycle, MFA, sharing links,
  phishing awareness, and what sensitive data employees paste into mail/drive.

One security risk if the model is misunderstood (write clearly)
---------------------------------------------------------------
Risk example — “IaaS treated like SaaS”:
  A team migrates a critical app to VMs and assumes “cloud is secure,” so
  they skip OS patching, leave default firewalls, and store DB passwords in
  plaintext on the disk. Attackers exploit an unpatched OS vulnerability.
  Impact: data breach, ransomware, exam-result leakage, regulatory failure.
  Root cause: misunderstanding Shared Responsibility — OS security on IaaS
  remains with the customer.

Other short risks you may mention:
  • Over-permissive IAM (“allAuthenticatedUsers” on a bucket)
  • Believing PaaS removes need for secure coding
  • Leaving SaaS sharing public without data classification

GKE note (optional bonus line)
------------------------------
On GKE, Google manages the control plane; customers still secure workloads,
IAM, images, and (on Standard) much of node configuration — a shared split.

Conclusion
----------
The Shared Responsibility Model assigns provider vs customer duties by
service model. IaaS gives customers the most OS/app burden; SaaS the least
— but data and identity duties almost always remain with the customer.
Misunderstanding this model creates preventable security incidents.

Q.3  Cloud-Native Architecture                                         [7]
--------------------------------------------------------------------------------

QUESTION RECALL
Define Cloud-Native Architecture. Explain microservices, containers,
managed services, API-driven design, externalized state. Differentiate from
monolith for scalability and failure handling.

MODEL ANSWER

Definition
----------
Cloud-Native Architecture is an approach to designing and running applications
that fully exploits cloud delivery models: elasticity, automation, managed
services, and resilience. It is NOT merely “hosting a traditional app on a
VM in someone else’s data centre.”

  Exam one-liner:
  “Cloud-native apps are built as loosely coupled services, packaged in
   containers, driven by APIs, with state kept outside the compute instance,
   so they can scale and recover using cloud primitives.”

Important characteristics (explain each with depth)
---------------------------------------------------

1. Microservices
   The application is decomposed into small, independently deployable
   services aligned to business capabilities (e.g., orders, products,
   payments, notifications). Each service can be developed, deployed, and
   scaled separately.
   Benefit: a spike in “orders” does not force you to scale the entire
   monolith, including rarely used admin modules.

2. Containers
   Each service (or process) is packaged with its dependencies into a
   container image (Docker). Containers start quickly and run consistently
   from laptop → staging → production.
   Often orchestrated by Kubernetes (Unit 4) for placement, restarts, and
   rollouts.

3. Managed services
   Prefer cloud-managed building blocks (Cloud SQL, Pub/Sub, object storage,
   managed Kubernetes, serverless) instead of operating every database and
   message broker yourself.
   Benefit: less undifferentiated heavy lifting; provider handles patching
   and HA of that managed layer (within the shared model).

4. API-driven design
   Services communicate through well-defined APIs (REST/gRPC), not through
   shared in-memory calls inside one process. APIs enable independent
   evolution, clear contracts, and gateway patterns.

5. Externalized state
   Compute instances and Pods are treated as disposable. Session data, files,
   and durable records live in external stores (Redis, Cloud SQL, GCS).
   If a VM/Pod dies, a new one can start and continue — no unique data was
   trapped on local disk.

Optional related ideas (short): CI/CD, infrastructure as code, 12-factor
config via environment variables, health checks, observability.

Cloud-native vs traditional monolith — SCALABILITY
--------------------------------------------------
  Monolith:
    • One deployable unit; scale usually means bigger VM or more copies of
      THE WHOLE app.
    • Hot module and cold module scale together → wasted capacity.
  Cloud-native:
    • Scale only the busy microservice (horizontal Pod/VM replicas).
    • Use load balancers, queues, and autoscaling per component.
    • Better resource efficiency and higher effective throughput under
      uneven load (e.g., dinner peak only on orders service).

Cloud-native vs traditional monolith — FAILURE HANDLING
-------------------------------------------------------
  Monolith:
    • Memory leak or crash in one module can take down the entire process.
    • Recovery often means restarting the whole application.
  Cloud-native:
    • Failure is isolated: payments can degrade while catalogue still serves.
    • Orchestrators restart unhealthy containers; LB removes bad backends.
    • Circuit breakers / retries / bulkheads (advanced) limit blast radius.

Diagram (draw)
--------------
  Monolith:   [ UI + Orders + Pay + Admin ] ----one process----
  Cloud-native:
       [UI] → API → [Orders] → [Pay]
                      ↓
                   [Cloud SQL / Pub/Sub]
       (each box scalable & restartable independently)

Real-life example
-----------------
Food-delivery / Fancy Store style system: frontend, orders, products as
separate services in containers. Dinner peak scales orders replicas; a bug
in notifications should not erase the ability to place orders if isolation
is designed well.

Conclusion
----------
Cloud-native architecture uses microservices, containers, managed services,
APIs, and externalized state so applications scale elastically and survive
component failure better than tightly coupled monoliths.

Q.4  Well-Architected Framework + scalable design practices            [7]
--------------------------------------------------------------------------------

QUESTION RECALL
Explain pillars Security, Reliability, Performance, Cost, Sustainability
with cloud examples. Also write best practices for designing scalable apps.

MODEL ANSWER

Introduction
------------
A Well-Architected Framework is a set of pillars / best-practice questions
used by cloud providers (AWS Well-Architected; Google Cloud Architecture
Framework has similar themes) to design systems that are secure, reliable,
efficient, cost-aware, and increasingly sustainable. For CC-II Unit 1 we
focus on the five pillars named in the syllabus.

1. Security
-----------
Protect data, systems, and assets; apply least privilege; defend in depth.
  Cloud examples:
    • IAM roles / service accounts with minimal permissions
    • VPC firewall rules and network tags
    • Encryption in transit (HTTPS) and at rest
    • Secret Manager instead of passwords in Git
    • Regular image / vulnerability scanning
  Teaching point: Shared Responsibility still applies — IAM mistakes are
  usually customer-side.

2. Reliability
--------------
Ability to recover from failures and meet availability targets.
  Cloud examples:
    • Multi-zone Managed Instance Groups behind HTTP(S) Load Balancing
    • Health checks and autohealing
    • Backups, PITR, and DR plans with RTO/RPO (Unit 2)
    • Kubernetes Deployments recreating crashed Pods
  Teaching point: design assuming instances WILL die.

3. Performance (Performance Efficiency)
---------------------------------------
Use resources efficiently to meet latency and throughput goals; evolve as
demand changes.
  Cloud examples:
    • Right-size machine types (e2-small vs e2-standard)
    • Autoscaling on CPU / load
    • Cloud CDN / caching for static content
    • Choose region close to users
    • Connection pooling / async messaging for spikes

4. Cost (Cost Optimization / FinOps thinking)
---------------------------------------------
Deliver business value at the lowest price point; avoid waste.
  Cloud examples:
    • Delete unused disks, IPs, old snapshots
    • Scale in after peaks; use Spot/Preemptible for batch
    • Prefer serverless / Autopilot when it reduces idle node cost
    • Budgets and billing alerts
  Teaching point: measured service (Q.1) makes cost visible — use that data.

5. Sustainability
-----------------
Minimise environmental impact of workloads.
  Cloud examples:
    • Autoscaling so idle VMs are not burning energy
    • Efficient regions / carbon-aware choices where policy allows
    • Right-sizing and managed services that improve utilisation
    • Batch jobs in off-peak windows when appropriate
  Teaching point: unused always-on capacity wastes both money and energy.

Best practices for designing SCALABLE cloud applications
--------------------------------------------------------
(Write as a numbered list — examiners like this)

  1. Design for horizontal scaling — add instances/Pods, not one giant server.
  2. Put a load balancer in front; only send traffic to healthy backends.
  3. Keep services loosely coupled (microservices + APIs + queues).
  4. Externalise state (DB, cache, object storage) — compute is disposable.
  5. Automate provisioning (IaC — Unit 3) and deployments (CI/CD — Unit 5).
  6. Implement health checks and readiness probes at every tier.
  7. Use autoscaling policies with sensible min/max bounds.
  8. Prefer managed services for databases/queues when they fit.
  9. Observe with metrics, logs, traces; alert on user-visible symptoms.
 10. Plan capacity for known peaks (result day / sale) and test beforehand.
 11. Use rolling updates so releases do not require full downtime.
 12. Apply least privilege and network segmentation as you scale out
     (more instances must not mean more open attack surface).

Integrated example
------------------
Campus admission portal on result week:
  Security — IAM for deployers; HTTPS at LB.
  Reliability — multi-zone MIG + DB backups.
  Performance — CDN for static pages; autoscale web tier.
  Cost — scale down after counselling week.
  Sustainability — no idle 50-server farm all year.

Conclusion
----------
The Well-Architected pillars give a checklist for quality cloud design;
scalable applications implement that checklist with horizontal scale, LB,
external state, automation, and continuous observation.

Q.5  Backup, Disaster, DR, BC — “Backup alone is not DR”               [7]
--------------------------------------------------------------------------------

QUESTION RECALL
Define Backup, Disaster, Disaster Recovery, Business Continuity. Justify:
“Backup alone is not Disaster Recovery,” with runbooks, failover/rebuild,
and restore testing.

MODEL ANSWER

Introduction
------------
Organisations confuse “we take backups” with “we can recover.” Unit 2
requires precise definitions and a clear justification of why backups are
only one ingredient of Disaster Recovery.

Core definitions (write as a table)
-----------------------------------
  TERM                 DEFINITION
  ─────────────────────────────────────────────────────────────────────────
  Backup               A secondary copy of data stored so it can be restored
                       after loss, corruption, deletion, or ransomware.
  Disaster             An event that makes IT systems unavailable or data
                       unusable for an unacceptable period (natural disaster,
                       zone outage, cyberattack, catastrophic bad release).
  Disaster Recovery    The documented and practised set of policies,
  (DR)                 processes, and technologies used to restore IT
                       services after a disaster to meet business goals.
  Business Continuity  The broader organisational capability to continue
  (BC)                 critical business functions for customers/staff —
                       includes people, communications, alternate processes,
                       not only IT restore.

  Related terms (optional but scoring):
    Restore  — bringing data/systems back to usable state.
    Failover — switching users to a secondary site/system.
    Failback — returning to primary after repair (when applicable).

Backup ≠ Disaster Recovery (central justification)
--------------------------------------------------
Having backup files answers: “Do we have a copy of the data?”
Disaster Recovery answers: “Can we restore SERVICE within agreed time,
with known people and steps, after a real failure?”

Therefore backup is NECESSARY but NOT SUFFICIENT for DR.

1) Runbooks (who does what, in what order)
------------------------------------------
A runbook is a step-by-step operating procedure:
  • Who is on-call? Who approves failover?
  • Which project / accounts / VPN are used?
  • Exact commands or IaC apply order
  • How to update DNS / Load Balancer
  • How to communicate status to users
Without a runbook, backups sit unused while teams argue during the outage.
DR requires decision rights and sequenced actions, not only files in a bucket.

2) Failover / rebuild procedures
--------------------------------
After a zone or region failure you may need to:
  • Promote a standby database
  • Rebuild VMs / GKE from Infrastructure as Code (Unit 3 link)
  • Attach restored disks / point apps at new endpoints
  • Shift traffic via DNS or load balancer
If infrastructure only existed as “click-ops memory,” restore of data alone
cannot recreate the full application path. DR needs rebuildable infra +
cutover steps.

3) Restore testing (prove it works)
-----------------------------------
Untested backups are false comfort. Regular DR drills / game days should:
  • Restore a backup to an isolated environment
  • Measure actual elapsed time
  • Compare against RTO (Unit Q.7)
  • Check data freshness against RPO
  • Fix gaps (missing configs, wrong IAM, outdated runbook)
Many outages reveal that backups were encrypted with a lost key, incomplete,
or too slow to restore.

Additional supporting arguments (use 2–3 lines)
-----------------------------------------------
  • Backups do not by themselves provide high availability during the outage.
  • Application consistency matters (DB PITR vs raw disk snapshot).
  • Geographic placement: backups only in the same burning zone do not help.

Real-life example
-----------------
A college keeps nightly VM snapshots. During a ransomware event, snapshots
exist, but:
  • nobody practised restore,
  • the runbook is outdated,
  • rebuilding the web tier takes 2 days of console work.
Result: data copy existed, but DR failed business needs. Contrast: tested
restore + Terraform rebuild + DNS failover meets a defined RTO.

Diagram
-------
  Backup files ──┐
  Runbooks ──────┼──► Successful Disaster Recovery
  Rebuild/Failover┤
  Tested restores┘

Conclusion
----------
Backup, Disaster, DR, and Business Continuity are related but different.
The statement “Backup alone is not Disaster Recovery” is justified because
DR also requires runbooks, failover/rebuild capability, and regular restore
testing that proves services return within business time limits.

Q.6  HA vs Fault Tolerance, SPOF, reducing SPOF                        [7]
--------------------------------------------------------------------------------

QUESTION RECALL
Differentiate HA and Fault Tolerance with cloud examples. Explain SPOF and
how redundancy, load balancing, health checks, and multi-zone reduce SPOF.

MODEL ANSWER

Introduction
------------
Cloud applications must survive component failure. Two related design goals
are High Availability and Fault Tolerance. Both fight the Single Point of
Failure (SPOF).

High Availability (HA)
----------------------
HA is the design goal of keeping a service reachable for users by minimising
downtime. It is often expressed as an uptime percentage.

  Example uptime intuition:
    99.9% ≈ about 8.8 hours downtime/year
    99.99% ≈ about 52 minutes downtime/year

  HA typically uses redundancy + automatic detection + failover so that when
  one instance fails, others continue serving.

Fault Tolerance (FT)
--------------------
Fault Tolerance is the ability to continue correct operation even when a
component fails — ideally without the user noticing an interruption.

  Relationship: HA systems are often built from fault-tolerant mechanisms
  (replicas, autohealing). Full DR across regions is a further step beyond
  basic HA inside one region.

Differentiation table
---------------------
  Aspect              High Availability                 Fault Tolerance
  ─────────────────────────────────────────────────────────────────────────
  Focus               Minimise downtime / keep service  Continue correctly
                      reachable                         despite fault
  User experience     Brief failover may be OK          Ideally seamless
  Typical pattern     LB + multiple instances           Active redundancy,
                                                        autohealing, retries
  Example             MIG of 3 VMs behind HTTP(S) LB    Autohealing replaces
                                                        unhealthy VM; K8s
                                                        recreates Pod

Cloud examples (expand)
-----------------------
  HA example: HTTP(S) Load Balancer distributes traffic to a Managed
  Instance Group with instances in multiple zones. If one VM fails, others
  still serve.
  FT building block: health-check failure triggers autohealing — a new VM
  is created to restore desired capacity without waiting for a human.

Single Point of Failure (SPOF)
------------------------------
A SPOF is any component whose single failure causes the entire service to
become unavailable.

  Classic SPOFs:
    • One web VM
    • One database with no replica/backup strategy
    • One availability zone only
    • One network path / one load balancer misconfigured as unique choke
      without redundancy where required
    • One engineer who alone knows the password (process SPOF)

How cloud patterns reduce SPOF
------------------------------

1. Redundancy
   Run more than one instance of critical components (N+1). Store multiple
   data copies. Prefer managed services with built-in replication where
   appropriate.
   Example: at least two application VMs; primary + standby DB.

2. Load balancing
   A load balancer spreads requests across healthy backends and provides a
   stable entry point. Clients do not depend on one VM’s IP.
   Example: Google Cloud HTTP(S) LB; Kubernetes Service / Ingress.

3. Health checks
   Periodic probes mark backends unhealthy so traffic stops going to dead
   instances. Combined with autohealing/autoscaling, capacity is restored.
   Example: HTTP health check on /health; Kubernetes readiness probes.

4. Multi-zone deployment
   Spread replicas across zones in a region so a single zone outage (power,
   networking) does not take down all copies.
   Example: regional MIG / regional GKE; dual-zone Cloud SQL HA config.

Diagram
-------
  Users → Load Balancer → VM1 (Zone A)
                       → VM2 (Zone B)
                       → VM3 (Zone C)
  Health checks remove bad VMs; redundancy keeps service up.

Real-life example
-----------------
UPI / payments API: multiple app replicas, multi-zone, LB, strict health
checks. A single Pod/VM failure must not stop checkout.

Conclusion
----------
HA minimises downtime; Fault Tolerance emphasises continued correct
operation through failures. SPOF is removed by combining redundancy, load
balancing, health checks, and multi-zone (and eventually multi-region DR
when the business requires it).

Q.7  RTO, RPO and DR strategies                                        [7]
--------------------------------------------------------------------------------

QUESTION RECALL
Define RTO and RPO with real-life examples. Explain Backup & Restore,
Pilot Light, Warm Standby, Hot Standby, Multi-site Active-Active. Compare
any two on cost, RTO, RPO.

MODEL ANSWER

Introduction
------------
RTO and RPO are the two business numbers that drive Disaster Recovery
design. Strategy choice is not about technology fashion — it is about
meeting these numbers at acceptable cost.

RTO — Recovery Time Objective
-----------------------------
Maximum acceptable DOWNTIME after a disaster — how long until the service
must be usable again.

  Looks FORWARD from the moment of disaster.
  Example:
    • UPI / payments: RTO = 5–15 minutes (money movement cannot wait).
    • University internal circulars site: RTO = 24 hours (tolerable).
    • Result portal on result day: RTO might be 1–2 hours (strict that day).

RPO — Recovery Point Objective
------------------------------
Maximum acceptable DATA LOSS, measured in time — how far back data may
roll when you restore.

  Looks BACKWARD from the moment of disaster to the last good copy.
  Example:
    • If RPO = 1 hour and disaster hits at 15:00, data written after ~14:00
      may be lost unless replication/PITR is tighter.
    • Hospital EHR may need RPO near zero; a blog may accept 24-hour RPO.

Diagram (must draw)
-------------------
  Last good backup/replica ----[ RPO window ]---- DISASTER ----[ RTO ]---- Restored
  • RPO = data-loss tolerance (time)
  • RTO = downtime tolerance (time)
  Do NOT swap the two terms in the exam.

How RTO/RPO drive cost
----------------------
  Lower RTO → need faster failover / more automation / more standby → cost ↑
  Lower RPO → need frequent backups or continuous replication → cost ↑
  Near-zero RTO + near-zero RPO ≈ Hot Standby or Active-Active (expensive).

DR strategies explained
-----------------------

1. Backup & Restore
   Periodic backups; on disaster, provision infra and restore data.
   Typical: higher RTO, RPO = backup interval. Lowest cost. Good for
   non-critical systems.

2. Pilot Light
   Minimal core always running in another region (e.g., DB replica / small
   core services). On disaster, scale out application tier around that core.
   Faster than cold backup; still needs scale-out time.

3. Warm Standby
   A scaled-down but complete environment always on; scale up on failover.
   Lower RTO than pilot light; medium cost.

4. Hot Standby (Active-Passive hot)
   Near-production capacity already running; failover is mostly traffic
   shift. Very low RTO; high cost because you pay for idle-ish capacity.

5. Multi-site Active-Active
   Two or more regions serve traffic simultaneously; data replication is
   carefully designed. Near-zero user downtime for many failures; highest
   complexity and cost.

Compare ANY TWO (write a full comparison — example Warm vs Hot)
---------------------------------------------------------------
  Criterion        Warm Standby                      Hot Standby
  ─────────────────────────────────────────────────────────────────────────
  Cost             Medium (smaller footprint)        High (near-full capacity)
  RTO              Low–medium (scale-up time)        Very low (traffic shift)
  RPO              Depends on replication/backup     Often very low with
                                                   continuous replication
  Ops complexity   Medium                            Higher (keep dual envs
                                                   healthy)
  Good for         Important portals with moderate   Payments, trading-like
                   budget                            critical APIs

(You may instead compare Backup & Restore vs Active-Active — state cost high

Q.8  Cloud backups, snapshots & one DR case study                      [7]
--------------------------------------------------------------------------------

QUESTION RECALL
Describe cloud backups/snapshots (disk/VM, object versioning, managed DB
PITR). Discuss one DR case study and justify strategy using RTO and RPO.

MODEL ANSWER

Introduction
------------
Cloud platforms provide multiple data-protection tools. Good answers name
the tool, what it protects, and its limits (crash-consistent vs application-
consistent, same-zone risk, etc.).

1. Disk / VM snapshots
----------------------
A snapshot is a point-in-time capture of a persistent disk (e.g., Compute
Engine snapshots). After the first snapshot, later ones are often
incremental (only changed blocks), which saves time and storage.

  Uses:
    • Rollback a VM disk after a bad patch
    • Clone environments
    • Part of backup strategy for stateful disks
  Limits:
    • Snapshot of a busy database disk may be crash-consistent, not always
      application-consistent — prefer managed DB backup/PITR for SQL.
    • Snapshots stored only in the same region/zone strategy may not cover
      regional disasters — check replication/geo options.

2. Object-storage versioning
----------------------------
Object stores (GCS / S3) can keep versions of objects. If a file is
overwritten or deleted, a previous version can be restored.

  Uses:
    • Protects against accidental delete/overwrite
    • Helps some ransomware scenarios if attacker cannot purge versions
      (combine with retention / immutable policies where available)
  Limits:
    • Versioning is not a full application DR plan by itself
    • Cost grows with many versions — lifecycle rules needed

3. Managed database backups & Point-in-Time Recovery (PITR)
----------------------------------------------------------
Managed databases (Cloud SQL, RDS, etc.) offer automated backups and often
PITR: restore to a precise timestamp within the retention window.

  Uses:
    • Tight RPO (minutes) compared with nightly-only dumps
    • Recover from user error (“DELETE without WHERE”) by restoring to
      just before the mistake
  Limits:
    • Still need a plan for WHERE to restore and how apps reconnect
    • PITR window is finite; test restores

Putting them together
---------------------
Production designs often COMBINE:
  continuous DB protection (HA + PITR) + object versioning + VM/app
  snapshots/IaC rebuild + cross-region copies for severe disasters.

Case study — Campus Result Portal (write as Problem → Approach → Justify)
-------------------------------------------------------------------------
Problem:
  On result day, the portal must stay available. Business sets:
    RTO ≤ 2 hours (service back same morning)
    RPO ≤ 15 minutes (marks data loss beyond 15 minutes unacceptable)
  Risks: zone outage, bad deployment, accidental DB delete.

Approach:
  • Cloud SQL with automated backups + PITR
  • Application VMs/MIG (or GKE) rebuildable with IaC
  • Disk snapshots for app templates if needed
  • Object versioning for published PDF/mark sheets in a bucket
  • Warm Standby or at least multi-zone HA + documented failover runbook
  • Quarterly restore test before exam season

Justification using RTO and RPO:
  • RPO ≤ 15 minutes: nightly backup alone is NOT enough (RPO could be 24h).
    PITR (and/or continuous replica) is required — this justifies managed DB
    PITR over “weekly zip files.”
  • RTO ≤ 2 hours: pure Backup & Restore of everything from zero may exceed
    2 hours. Warm Standby / multi-zone HA + IaC rebuild is justified to meet
    RTO without paying for full Active-Active all year.
  • Snapshots/versioning support quick rollback of app/content mistakes that
    are not full regional disasters.

Conclusion
----------
Cloud DR uses snapshots, object versioning, and managed DB backups/PITR as
complementary tools. A strong 7-mark case study states RTO/RPO first, then
selects a strategy and tools that mathematically satisfy those numbers.

Q.9  What is IaC? Why preferred? Benefits + scenario                   [7]
--------------------------------------------------------------------------------

QUESTION RECALL
Define IaC. Why preferred over console provisioning? Benefits + one
practical scenario where IaC is highly useful.

MODEL ANSWER

Definition
----------
Infrastructure as Code (IaC) is the practice of managing and provisioning
computing infrastructure — VPCs, subnets, firewalls, VMs, load balancers,
clusters, buckets, IAM bindings, and more — through machine-readable
definition files, rather than through manual console clicking or one-off
unreproducible commands.

  Exam one-liner:
  “IaC means treating infrastructure the same way we treat application code
   — versioned, reviewed, tested, and automated.”

What counts as infrastructure here
----------------------------------
Not only VMs: networks, storage, databases, DNS, autoscaling policies,
alert policies — anything creatable via cloud APIs can usually be declared.

Traditional (manual) way vs IaC way
-----------------------------------
  Manual / console (click-ops):
    Open console → Create VM → remember 15 settings → repeat for Dev/Prod
    → hope Prod matches Dev → after disaster, try to remember what existed.

  IaC way:
    Write a file (Terraform HCL / CloudFormation YAML / etc.) → store in Git
    → review → plan → apply → cloud resources match the file.
    Same file recreates Dev, Staging, Prod shapes consistently.

Why IaC is preferred over manual console provisioning
-----------------------------------------------------
  1. Reproducibility — environments can be recreated identically.
  2. Reviewability — pull requests catch open SSH or costly oversized VMs.
  3. Speed at scale — 50 resources are not 50 human click paths.
  4. Disaster recovery support — rebuild from code (links Unit 2 RTO).
  5. Audit trail — Git history shows who changed what and why.
  6. Reduced drift — desired state documented; console hotfixes discouraged.
  7. Automation — CI pipelines can run plan on every change.

Benefits of IaC (expand for marks)
----------------------------------
  • Repeatability — same definition → same environment shape
  • Version control — branch, diff, blame, revert
  • Consistency — Dev ≈ Staging ≈ Prod (parameterised differences only)
  • Fewer human errors — less missed firewall rules
  • Faster onboarding — new engineer reads the repo, not tribal knowledge
  • Cost visibility — instance sizes are explicit in code
  • Safer changes — plan shows destroys before they happen
  • Collaboration — infra changes follow the same PR culture as app code

Practical scenario (detailed — highly useful)
---------------------------------------------
Scenario: University result portal scale-out
  Before IaC: on result morning, two engineers click in the console to create
  extra VMs, attach them to a load balancer, open firewall tags — under
  stress, one forgets health checks; traffic blackholes.

  With IaC: `instance_count` or autoscaling parameters live in Terraform.
  Engineer opens PR → plan shows +6 instances → review → apply → verify.
  After results week, apply again with smaller count → cost drops.

  Why IaC is “highly useful” here: peak is predictable but stressful; code
  changes are calmer and repeatable than emergency click-ops; also supports
  rebuilding if a zone fails (DR).

Anti-patterns to mention briefly
--------------------------------
  • Click in console and never update code → permanent drift
  • Secrets committed into Git
  • terraform apply to prod with no plan review

Conclusion
----------
IaC replaces fragile manual provisioning with versioned, reviewable
definitions. It is preferred because it improves speed, safety, consistency,
and DR rebuild capability — especially visible in peak-scale scenarios like
a result portal.

Declarative vs task-oriented (explain for marks)

    terraform init
    terraform plan    # preview: + create bucket
    terraform apply   # creates bucket
    terraform apply   # second run: no changes (idempotent desired state)

Q.11  Configuration Management vs provisioning; Ansible steps          [7]
--------------------------------------------------------------------------------

QUESTION RECALL
What is Configuration Management? Differentiate provisioning vs software
configuration. Explain Ansible (or equivalent) for CM and typical automated
server setup steps.

MODEL ANSWER

Definition — Configuration Management (CM)
------------------------------------------
Configuration Management is the process of maintaining computer systems,
software, and settings in a known, desired state over time. In cloud labs,
CM usually means: packages installed, config files correct, services running,
users present — repeatedly and safely.

Infrastructure provisioning vs software configuration
-----------------------------------------------------
  Aspect                    Provisioning                    Configuration Mgmt
  ─────────────────────────────────────────────────────────────────────────────
  Question answered         “Does the cloud resource        “Is the software on
                            exist with correct shape?”       that resource set up
                                                             correctly?”
  Examples                  VPC, subnet, VM, LB,            nginx installed,
                            Cloud SQL instance, GKE         app user created,
                            cluster                         env file placed,
                                                            service enabled
  Typical tools             Terraform, CloudFormation       Ansible, Puppet,
                                                            Chef, startup scripts
  Metaphor                  Build the house                 Furnish the rooms
  Cloud API focus           Create/update cloud objects     Change guest OS / app
  Unit link                 Unit 3 core                     Unit 3 + lab startup
                                                            scripts parallel

Why the split matters
---------------------
Creating a blank VM (provisioning) does not make your application ready.
Without CM, engineers SSH and configure by hand → snowflake servers → DR
and scaling become painful.

How Ansible is used for configuration management
------------------------------------------------
  • Language: YAML playbooks describing tasks and desired module states
  • Agentless: control node pushes over SSH (no agent required on target)
  • Inventory: list of hosts / groups (web, db)
  • Modules: apt/yum, copy, template, service, user, etc.
  • Idempotence goal: re-running the playbook should converge to the same
    state without harm (package state=present, not blind reinstall loops)
  • Use cases: bootstrap web servers, harden SSH, deploy app artifacts,
    manage Supervisor/systemd units

Typical steps in automated server setup (write all 8)
-----------------------------------------------------
  1. Update package index / install base OS packages
  2. Install application runtime (Node.js, Java, Python, nginx, …)
  3. Create a dedicated non-root application user
  4. Pull or copy application artifacts (from bucket/Git/CI)
  5. Write configuration files and environment variables securely
  6. Configure a process manager (systemd / Supervisor) for restart policy
  7. Start and enable services on boot
  8. Verify health (curl localhost:PORT/health, check logs)

Lab parallel (Fancy Store / startup script idea)
-----------------------------------------------
A Compute Engine startup script that apt-installs tools, gsutil-copies the
app from a bucket, and configures Supervisor is essentially an imperative
CM procedure. Ansible makes that procedure reusable, idempotent, and
reviewable in Git.

Images vs bootstrap (short enhancement)
---------------------------------------
  • Bootstrap at boot: generic image + Ansible/startup on each boot
  • Golden image: bake packages into a custom image for faster scale-out
Both are CM strategies; containers (Unit 4) push packing further.

Conclusion
----------
Configuration Management keeps software state desired and consistent.
Provisioning creates cloud resources; CM configures them. Ansible provides
agentless, YAML-driven automation for repeatable server setup steps that
exams expect you to list clearly.

Q.12  IaC workflow + versioning/scaling + plan-before-apply            [7]
--------------------------------------------------------------------------------

QUESTION RECALL
Explain Write → Version Control → Review → Plan → Apply → Verify. Discuss
best practices for versioning and scaling. Why is Plan before Apply important?

MODEL ANSWER

Introduction
------------
IaC is not only “having a .tf file.” Marks are awarded for the operational
workflow that keeps changes safe in shared cloud environments.

The standard workflow (explain EACH step)
-----------------------------------------

1. Write
   Author infrastructure definitions (Terraform HCL, CloudFormation
   templates, or related modules). Capture intent: network, compute, IAM,
   sizes, tags/labels.

2. Version Control
   Commit to Git on a feature branch. Git becomes the source of truth.
   History supports audit, blame, and revert. Never rely on console as the
   undocumented blueprint.

3. Review
   Open a pull request. Peers check for:
     • 0.0.0.0/0 on sensitive ports
     • accidental database destruction
     • oversized costly machines
     • missing tags / backups
   Review is a human safety layer before the cloud is touched.

4. Plan
   Run terraform plan or create a CloudFormation change set.
   The tool previews: resources to add, change, destroy.
   This is the critical “diff before impact” step.

5. Apply
   Execute the approved change (terraform apply / stack update), ideally
   from a controlled pipeline with credentials of limited scope.

6. Verify
   Confirm reality matches intent: health checks, smoke tests, dashboards,
   kubectl get / curl, IAM spot checks. Roll back or follow-up commit if
   verification fails.

Diagram
-------
  Write → Git → PR Review → Plan → Apply → Verify → (feedback to Write)

Best practices for VERSIONING infrastructure
--------------------------------------------
  • Git is source of truth; discourage long-lived console drift
  • Use branches + protected main/prod branches
  • Tag releases; pin provider and module versions
  • Use remote state + locking (Terraform) for teams
  • Separate variables for Dev/Staging/Prod; shared modules
  • NEVER commit production secrets; use Secret Manager / CI secrets
  • Code review required for prod-impacting changes
  • Document module inputs/outputs for reuse

Best practices for SCALING infrastructure with IaC
--------------------------------------------------
  • Parameterise counts: `instance_count`, min/max replicas
  • Express autoscaling policies in code (MIG utilisation, HPA later)
  • Promote the same modules Dev → Staging → Prod
  • Label/tag resources for cost allocation as you scale
  • Scale DOWN in code after peaks (FinOps)
  • Pair with DR: capacity parameters for Warm/Hot standby environments

Why “Plan before Apply” is important (high-value paragraph)
-----------------------------------------------------------
Plan reveals unintended consequences BEFORE they damage production.
Classic example: renaming a database resource may show destroy+create —
which would wipe data if applied blindly. Plan also surfaces unexpected
IAM or firewall diffs from drift. In shared environments, applying without
plan review is an anti-pattern because one engineer’s laptop apply can
outage an entire class of users. Plan turns infrastructure changes into
predictable, reviewable operations — essential for exams and for real ops.

Real-life example
-----------------
E-commerce festival sale: PR increases MIG max replicas; plan shows only
scaling changes; apply after approval; verify latency. Without plan, a
mistyped resource name might recreate a DB mid-sale.

Conclusion
----------
The IaC workflow Write → Version → Review → Plan → Apply → Verify makes
infrastructure changes safe and collaborative. Versioning and parameterised
scaling keep environments consistent; Plan-before-Apply is the key control
that prevents accidental destruction and costly mistakes.

Q.13  Kubernetes definition, need, architecture, Pod/Deploy/Service    [7]
--------------------------------------------------------------------------------

QUESTION RECALL
What is Kubernetes? Why is orchestration needed? Describe control plane
(API Server, Scheduler, Controller Manager, etcd) and worker components
(kubelet, kube-proxy, runtime). Define Pod, Deployment, Service.

MODEL ANSWER

Introduction / Definition
-------------------------
Kubernetes (K8s) is an open-source container orchestration platform that
automates the deployment, scaling, networking, and management of
containerized applications across a cluster of machines.

  Exam one-liners:
  • “Kubernetes is the operating system for your containers at cluster scale.”
  • Origin: inspired by Google Borg; now maintained under CNCF.

Why container orchestration is needed
-------------------------------------
Containers (e.g., Docker images) package applications consistently. Running
ONE container on ONE machine is easy. Problems appear at fleet scale:

  Without an orchestrator you must manually:
    • Decide which server runs which container (placement)
    • Restart containers when they crash (healing)
    • Add/remove replicas before/after traffic spikes (scaling)
    • Route users only to healthy instances (service discovery / LB)
    • Roll out new versions without long downtime (updates)

  Manual SSH and ad-hoc scripts become error-prone, slow, and non-uniform.
  Kubernetes automates these operational loops using controllers that
  continuously reconcile desired state with actual state.

Architecture overview
---------------------
A Kubernetes cluster has two major parts:
  1) Control plane — the “brain” (decisions, API, cluster state)
  2) Worker nodes — the “muscle” (where containers actually run)

Diagram (draw this for marks)
-----------------------------
  ┌──────────────── CONTROL PLANE ────────────────┐
  │  API Server   Scheduler   Controller Manager  │
  │                   etcd                        │
  └─────────────────────┬─────────────────────────┘
                        │ schedules Pods
       ┌────────────────┼────────────────┐
       ▼                ▼                ▼
    Node 1           Node 2           Node 3
   kubelet          kubelet          kubelet
   kube-proxy       kube-proxy       kube-proxy
   runtime          runtime          runtime
   [Pod][Pod]       [Pod]            [Pod][Pod]

Control-plane components (explain each)
---------------------------------------
1. API Server
   Front door of the cluster. All tools (kubectl, controllers, UI) talk to
   the API Server. It validates and processes REST requests and updates
   cluster state.

2. Scheduler
   Watches for Pods that are not yet assigned to a node and selects a
   suitable worker node based on resource needs, constraints, and policies.

3. Controller Manager
   Runs control loops (Deployment, ReplicaSet, Node controllers, etc.).
   Each loop compares desired state vs actual state and acts to reconcile
   (e.g., create a replacement Pod if one crashes).

4. etcd
   Highly available key-value store holding cluster configuration and state.
   If etcd is unhealthy, cluster control plane health is at risk — hence
   managed Kubernetes (GKE) operates this carefully for you.

Worker-node components (explain each)
-------------------------------------
1. kubelet
   Agent on every node. Ensures containers described in PodSpecs are running
   and healthy as instructed by the control plane.

2. kube-proxy
   Helps implement Kubernetes Service networking / load-balancing rules so
   that Services can reach Pods across the cluster.

3. Container runtime
   Software that actually pulls images and starts containers (containerd,
   historically Docker). Kubernetes schedules; the runtime executes.

Desired state idea (link for extra clarity)
-------------------------------------------
You declare “3 replicas of hello-app.” Controllers create/replace Pods until
reality matches. This self-healing behaviour is a core exam phrase:
  “Desired state reconciled by controllers.”

Definitions — Pod, Deployment, Service
--------------------------------------
  Pod
    Smallest deployable unit in Kubernetes. One or more tightly coupled
    containers sharing network namespace (same localhost/IP inside the Pod)
    and storage volumes. Pods are mortal; IPs change when Pods are replaced.

  Deployment
    Higher-level object that declares a Pod template and a desired replica
    count. Manages ReplicaSets and supports rolling updates and rollbacks
    so releases can proceed with near-zero downtime.

  Service
    Stable virtual IP / DNS name that selects Pods using labels. Provides a
    consistent endpoint even as Pods are created and destroyed. Types
    include ClusterIP, NodePort, LoadBalancer (detail in later topics).

Real-life parallel
------------------
Food-delivery microservices: many Pods for orders/payments; control plane
places and restarts them; Services give stable names between services.

Conclusion
----------
Kubernetes orchestrates containers at cluster scale. The control plane
(API Server, Scheduler, Controller Manager, etcd) decides and stores state;
workers (kubelet, kube-proxy, runtime) run Pods. Pod, Deployment, and
Service are the essential objects for explaining how apps are packaged,
replicated, and reached.

Q.14  GKE setup & management — Autopilot vs Standard                   [7]
--------------------------------------------------------------------------------

QUESTION RECALL
Explain steps to set up/manage a GKE cluster. Compare Autopilot vs Standard.
Describe kubectl connection, namespaces, and basic management (channels,
drain/cordon).

MODEL ANSWER

Introduction
------------
Google Kubernetes Engine (GKE) is Google Cloud’s managed Kubernetes service.
Google runs the control plane; you focus on workloads (and, depending on
mode, nodes). This question is Practical-4 oriented: setup, connect, manage.

End-to-end setup steps (write in order)
---------------------------------------
  1. Choose a GCP project with billing enabled; select region.
  2. Enable required APIs:
       • Compute Engine API
       • Kubernetes Engine API
       • Artifact Registry API (for later image pulls)
  3. Create the cluster:
       • Autopilot: gcloud container clusters create-auto NAME --region=REGION
       • Standard:  gcloud container clusters create NAME --num-nodes=N …
  4. Connect kubectl credentials:
       gcloud container clusters get-credentials NAME --region=REGION
  5. Verify cluster access:
       kubectl get nodes
       kubectl cluster-info
       kubectl get namespaces
  6. Organise with namespaces (dev/staging/prod or per team).
  7. Grant the node / GKE service account permission to pull images from
     Artifact Registry (needed before real deploys).
  8. Apply baseline security: least-privilege IAM for who may get-credentials.

Creation takes several minutes while control plane and nodes become Ready.

Autopilot vs Standard (full comparison)
---------------------------------------
  Aspect                 Autopilot                         Standard
  ─────────────────────────────────────────────────────────────────────────
  Control plane          Google-managed                    Google-managed
  Node management        Google manages nodes              You create/size
                                                         node pools
  Billing model          Pay for Pod resource requests     Pay for node VMs
  Operational effort     Lower                             Higher
  Flexibility            Opinionated safe defaults         GPUs, custom
                                                         machine types, more
                                                         low-level control
  Scaling nodes          Google adds capacity as Pods need Cluster Autoscaler /
                                                         manual pools
  Best fit               Most labs & many production apps  Special hardware /
                                                         unusual node needs
  Example command        create-auto …                     create … --num-nodes

  Exam line:
  “Both give a managed control plane; Autopilot also manages nodes and bills
   per Pod requests; Standard gives you node-pool control and VM billing.”

Connecting with kubectl (what get-credentials does)
---------------------------------------------------
get-credentials writes/updates your kubeconfig so the kubectl CLI targets
that cluster’s API Server. After this, ordinary Kubernetes commands work:

  kubectl get pods -A
  kubectl apply -f deployment.yaml
  kubectl describe node NODE

All management traffic goes to the API Server (Unit Q.13 architecture).

Role of namespaces
------------------
Namespaces are virtual clusters inside one physical cluster. They isolate
resources by environment or team without paying for many control planes.

  Examples:
    kubectl create namespace staging
    kubectl create namespace production
    kubectl apply -f app.yaml -n staging

  Campus lab pattern: one shared GKE cluster; each student team gets a
  namespace; faculty retains cluster-admin.

Basic cluster management practices
----------------------------------

1. Release channels & upgrades
   GKE can auto-upgrade via channels:
     Rapid   — newest features (testing)
     Regular — balance of features and stability (common default)
     Stable  — slowest, most conservative
   Prefer regional clusters for higher control-plane availability.
   Set maintenance windows so upgrades do not surprise exam/result day.
   Test workloads on non-prod before production upgrades.

2. Drain & cordon (node operations — especially Standard)
   • kubectl cordon NODE  — mark unschedulable (no new Pods)
   • kubectl drain NODE --ignore-daemonsets — evict Pods for maintenance
   • kubectl uncordon NODE — allow scheduling again
   Autopilot: Google handles much node repair/replacement for you.

3. Ongoing hygiene
   • Watch node NotReady / Pod Pending (capacity)
   • Keep kubectl client reasonably close to cluster version
   • Limit who can get-credentials (IAM least privilege)
   • Prefer Workload Identity over downloading long-lived key JSON
   • Optional IaC link (Unit 3): create GKE with Terraform for repeatability

Real-life example
-----------------
Semester cloud lab: Autopilot cluster, Regular channel, namespaces per team,
maintenance window on Sunday night, faculty-only cluster-admin.

Conclusion
----------
GKE setup follows enable APIs → create cluster → get-credentials → verify.
Autopilot trades control for simplicity and Pod-based billing; Standard
offers node-level control. Namespaces organise workloads; release channels
and drain/cordon practices keep clusters safely managed over the semester.
